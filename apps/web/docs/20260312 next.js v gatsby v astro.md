Ok, so actually I'm questioning the use of Astro, it's just not built for what we're doing. The useful aspects of it have alternatives in Gatsby and Next. 

Gatsby would be perfect if it wasnt no longer being properly supported, after Netlify bought it, they laid off the entire team. For a framework with over 100 depedeneices, that can make things messy. Also there's apparently super long build times which will make dev a pain in the ass. 

PostHog choose Gatsby before the decline of the framework. 

Even though the main part of our site is a client:only island, that doesnt neccessarily make Astro reduant, we're still using it to manage our markdown content, give layouts and to provide SEO friendly individual pages. 

Another alternative is Next.JS: I feel like this might be the one?

> # When I asked Claude
> 
> ## Where Next.js genuinely wins
> 
> ### 1. Architectural honesty
> 
> Your app is fundamentally a hybrid — static content pages plus a heavy interactive client. Next.js is designed for exactly this without any philosophical tension. No `client:only` workarounds, no island boundaries, just React throughout with per-route rendering strategies:
> 
> ```
> app/
> ├── page.tsx              ← the OS desktop (client component)
> ├── posts/[slug]/
> │   └── page.tsx          ← static, SEO optimised (server component)
> └── tags/[tag]/
>     └── page.tsx          ← static, SEO optimised (server component)
> ```
> 
> Each route declares its own rendering strategy naturally.
> 
> ### 2. Persistent layout shell
> 
> Next.js App Router layouts **genuinely don't remount** between navigations. This means your OS chrome, window state, and graph state all survive route changes natively — no URL param tricks needed to persist state across navigation. In Astro, the `?post=slug` workaround exists precisely because Astro doesn't have this.
> 
> ### 3. Cleaner content fetching
> 
> Instead of Devin's `fetch()` + `DOMParser` hack to extract `<main>` from a fully rendered page, Next.js lets you fetch raw markdown or a JSON endpoint and render it properly inside the window:
> 
> ts
> 
> ```ts
> // Clean — fetch just the data you need
> const post = await fetch(`/api/posts/${slug}`).then(r => r.json())
> ```
> 
> No scraping your own HTML.
> 
> ### 4. API routes built in
> 
> You can expose `/api/posts/[slug]` as a proper JSON endpoint — returning frontmatter, rendered HTML, and metadata cleanly. This makes the window content fetching robust and explicit rather than parsing rendered pages.
> 
> ### 5. Incremental Static Regeneration (ISR)
> 
> For a huge markdown corpus, ISR means you don't rebuild every page on every deploy. Pages regenerate on demand when content changes — Astro has no equivalent and full rebuilds get slow at scale.
> 
> ### 6. TypeScript integration
> 
> Next.js has deeper TypeScript integration out of the box — typed route params, typed `fetch` responses via server components, and better inference throughout. Given the existing codebase already has `any` type problems, a stricter foundation helps.
>

## Where it doesn't change much

| Concern                  | Reality                                                            |
| ------------------------ | ------------------------------------------------------------------ |
| SEO                      | Both produce static HTML for post pages — no meaningful difference |
| D3 + react-rnd + Zustand | Identical in both — these are React libraries, framework agnostic  |
| Markdown processing      | Both handle it well, just different APIs                           |
| Deployment on Vercel     | Both are first-class citizens                                      |


## The single biggest practical benefit

The `fetch()` + `DOMParser` approach Devin built is a workaround for a fundamental Astro limitation — you're scraping your own rendered pages to get content into a React component. In Next.js this entire problem disappears because your content and your React components live in the same world natively. **That alone removes a significant source of future bugs and complexity.**

## So . . .

The migration cost will be annoying, but I think Next.Js removes several awkward things that will otherwise accumulate into real maintenance burden.

The URL approach described for Astro in [20260310 Website Scope](20260310%20Website%20Scope.md) also will just really struggle with multiple windows I imagine. 

# The other requirements

We also want a live database connected app (the mDot platform) hosted on the site at some point.  This means:

- Runtime data fetching (not build-time static JSON)
- Authentication
- Real-time updates potentially
- API layer between database and frontend

Astro has no native server runtime for this. You'd need separate backend infrastructure alongside it - which works but adds complexity. Next.js **natively handles this** via Server Components, Server Actions, and API routes that run at request time against a live database.

We want the app to exist on the site, and on IOS and Android as standalone apps. 

Next.JS is a react framework, Astro is a web/javascript framework. That means Next can be used meaingfully for the App, while Astro cannot. For cross platform app dev, react is the most popular behind native (Kotlin/Swift) languages. 

By keeping with Astro, we'll end up running Astro alongside a separate backend, adding infrastructure complexity with no real benefit.
